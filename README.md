# Kafka Real-Time Orders Pipeline

## Project Overview

This project demonstrates a **production-style event-driven data pipeline** built using **Apache Kafka and Python**.
The system simulates a real-world retail platform where customer orders are generated continuously and processed in real time.

The pipeline ingests order events, processes them through Kafka consumers, handles failures using a Dead Letter Queue (DLQ), and stores the processed data in a data lake format suitable for analytics.

The primary goal of this project is to demonstrate how modern **streaming architectures** are designed to handle **high-throughput, fault-tolerant, scalable data pipelines**.

---

## Problem Statement

Traditional batch ETL systems process data periodically, which introduces delays and limits scalability. Modern platforms require **real-time ingestion and processing of events** such as transactions, logs, and IoT data.

This project simulates a retail system where orders are continuously generated and must be:

* Ingested in real time
* Processed reliably
* Stored in a structured format
* Designed to handle failures and scaling

---

## Architecture Overview

The pipeline consists of multiple components working together to enable real-time data streaming.

Data flow:

Order Generator → Kafka Producer → Kafka Topic → Kafka Consumer Group → Processing Layer → Data Storage

### Components

**Order Generator**

Simulates incoming retail orders.

**Kafka Producer**

Publishes order events to Kafka topics.

**Kafka Cluster**

Acts as a distributed event streaming platform that stores and distributes messages.

**Kafka Consumer Group**

Consumes events from Kafka topics in parallel for processing.

**Processing Layer**

Validates and transforms events.

**Storage Layer**

Writes processed data into partitioned files for analytical workloads.

**Dead Letter Queue (DLQ)**

Invalid or failed events are redirected to a separate Kafka topic for later inspection and reprocessing.

---

## Key Concepts Demonstrated

This project implements several important streaming architecture concepts.

**Event Driven Architecture**

Systems communicate through events rather than direct service calls.

**Kafka Topics and Partitions**

Events are organized into topics and distributed across partitions for scalability.

**Consumer Groups**

Multiple consumers process messages in parallel while maintaining fault tolerance.

**Offset Management**

Consumers commit offsets only after successful processing to prevent data loss.

**Failure Handling**

Messages that cannot be processed successfully are redirected to a Dead Letter Queue.

**Data Lake Integration**

Processed events are stored as Parquet files to enable downstream analytics.

---

## Expected Pipeline Flow

1. Order events are generated by the producer service.
2. The producer publishes events to a Kafka topic.
3. Kafka stores the events in partitions.
4. Consumer services read events from Kafka.
5. Events are validated and processed.
6. Successful events are written to storage.
7. Failed events are routed to a Dead Letter Queue.
